調査結果:各コード方式の実装可能性
前提として、このアプリは「独自エンコーダ + 独自デコーダ(QR系のみ)」または「独自エンコーダ + ZXing任せ(その他)」という構成です。ZXingが元々デコード対応している方式は、エンコーダさえ書けばデコードは無料でついてくるため、これが実装優先度を大きく左右します。

総合的な提案
実装コストと効果のバランスを考えると、優先順位は以下のようになります:

最優先(低コスト・ZXingデコード連携あり): UPC-A/UPC-E, Code 93, Codabar, ITF/ITF-14, Industrial・Matrix 2 of 5, MSI, Plessey, Pharmacode, Code 11 — どれも既存のCode39/Code128実装のパターンを流用でき、デコードもZXingに乗るので「往復対応」まで到達しやすい
中期候補(ZXingデコードあり・固定サイズで実装単純): MaxiCode — 唯一の2次元コードで、デコードがタダでついてくる点が大きい
要検討(自前デコードが前提、工数大): Han Xin Code, DotCode, Telepen — 生成のみ(現状のGS1 DataBar-14と同様「デコード不可」注記付き)で妥協するなら現実的
非推奨: JAB Code(アーキテクチャ的に別物), Boxing barcode(仕様不明), ShotCode(廃れた技術)
まずは「1」のグループから着手するのが費用対効果が高いと思いますが、いかがでしょうか。

実装結果
12種類のうち、実際にZXing(既存ライブラリ)でのデコードにより検証できたものを中心に実装しました。

✅ 実装・検証済み(ZXingデコードで往復確認)
UPC-A — 実際のGTIN例(03600029145)で符号化→ZXingデコードが完全一致
Code 93 — チェックディジット(C/K)込みで往復確認
ITF (Interleaved 2 of 5) — 往復確認
ITF-14 — 往復確認(GS1チェックディジット込み)
✅ 実装済み(アルゴリズムが自明で表テーブルに依存しない)
Industrial 2 of 5 — 標準的な2-of-5パターンで実装(ZXingは非対応のためデコード検証はできませんが、往復デコードの必要がない単純な重み付け構造です)
Pharmacode — 2進表現そのものなので表テーブル自体が存在せず、実装リスクは低いです
⚠️ 実装したが未検証(要注意)
UPC-E・Codabar — 一通り実装しましたが、ZXingのデコーダで正しく読み取れることを確認できませんでした。UPC-Eはパリティテーブルと開始/終了ガードパターンの全組み合わせ(384通り)を総当たりで試しても一致せず、Codabarも同様に失敗しました。原因は「私の記憶によるテーブルの誤り」か「このzxing-jsビルド自体がUPC-E/Codabarデコードを実質サポートしていない」のいずれか切り分けられませんでした(ZXing自身のエンコーダもこの2方式には非対応というヒントもあり、後者の可能性もあります)。この2つはUIには反映せず、コードのみ残してあります。信頼できる別ツールでの動作確認を推奨します。
❌ 今回見送り
Code 11 / MSI / Plessey / Matrix 2 of 5 — ZXingによる検証手段がなく、かつ幅パターンの完全な記憶に自信が持てなかったため、誤ったバーコードを実装として提示するリスクを避けて見送りました
Aztec Rune / CODABLOCK F / PDF417 — 今回のスコープには含めておらず未着手です(それぞれ別途取り組む価値はあります)
UI反映: バーコード規格タブに「UPC-A / Code 93 / ITF / ITF-14 / Industrial 2 of 5 / Pharmacode」が追加され、PNG書き出し・デコード連携(UPC-A/Code93/ITFはZXing経由で読み取り対応)まで一通り動作確認済みです。コンソール/ページエラーもありません。

未実装・未検証分について、優先的に取り組んでほしいものがあれば教えてください。

修正したバグ(7件)
機能が壊れていたもの

編集時に復号結果が更新されない — モジュールをクリック反転しても、復号結果のキャッシュ(_decodedCache)が無効化されず「内容」表示が古いままでした。SPEC の「リアルタイム復号」が実質壊れていた状態です(app.js)。
編集クリックの座標ズレ — 画面に表示されるのは canvas ではなく qr-card 全面に拡大された #qr-image なのに、クリック座標を canvas の矩形で換算していたため、表示域の端ほど数モジュール分ずれた場所が反転していました。img の矩形基準の換算に修正(app.js:1512)。
絵文字が分割で壊れる — 容量超過時の splitEvenly が UTF-16 コード単位で slice しており、サロゲートペア(絵文字等)の途中で分断されると文字化けしたコードが生成されていました。コードポイント単位に修正。
規格非準拠だったもの(実機スキャナで読めない原因)

Aztec Rune — ISO/IEC 24778 Annex A が要求する「モードメッセージの交互ビット反転」が未実装で、例えば値0のRuneは通常のコンパクトAztecと物理的に区別不能でした。エンコード/デコード両方に反転を追加(往復互換は維持)。
Industrial 2 of 5 — 数字間ギャップが省略されバー同士が融合(幅4のバーが出現)し、さらにスタート/ストップが ITF 用のものでした。規格通りのスタート(太太狭)・ストップ(太狭太)・全バー間狭スペース構成に修正。dev.md の「たまに読めないバーコードがある」はおそらくこれです。
その他

PDF417 の設定がシェアリンクに入らない + URLパラメータが無検証(mask=99 等で壊れたQRが黙って表示される、色に任意文字列が注入できる)— PDF417 の誤り訂正/列数を永続化対象に追加し、全パラメータに範囲検証を追加。
バーコードプレビューの高さ制約が 16*dpr と 16*scale で単位不一致(高DPI以外でブロックが余白へはみ出す)。

スキャン結果の色反映(3箇所)と色UI同期(5箇所)の重複を applyScanColors / syncColorInputs に共通化。
BARCODE_GROUPS をモジュールスコープへ移動(毎回の再生成を解消、symbology 検証にも再利用)。
