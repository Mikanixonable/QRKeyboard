調査結果:各コード方式の実装可能性
前提として、このアプリは「独自エンコーダ + 独自デコーダ(QR系のみ)」または「独自エンコーダ + ZXing任せ(その他)」という構成です。ZXingが元々デコード対応している方式は、エンコーダさえ書けばデコードは無料でついてくるため、これが実装優先度を大きく左右します。

総合的な提案
実装コストと効果のバランスを考えると、優先順位は以下のようになります:

最優先(低コスト・ZXingデコード連携あり): UPC-A/UPC-E, Code 93, Codabar, ITF/ITF-14, Industrial・Matrix 2 of 5, MSI, Plessey, Pharmacode, Code 11 — どれも既存のCode39/Code128実装のパターンを流用でき、デコードもZXingに乗るので「往復対応」まで到達しやすい
中期候補(ZXingデコードあり・固定サイズで実装単純): MaxiCode — 唯一の2次元コードで、デコードがタダでついてくる点が大きい
要検討(自前デコードが前提、工数大): Han Xin Code, DotCode, Telepen — 生成のみ(現状のGS1 DataBar-14と同様「デコード不可」注記付き)で妥協するなら現実的
非推奨: JAB Code(アーキテクチャ的に別物), Boxing barcode(仕様不明), ShotCode(廃れた技術)
まずは「1」のグループから着手するのが費用対効果が高いと思いますが、いかがでしょうか。

12種類のうち、実際にZXing(既存ライブラリ)でのデコードにより検証できたものを中心に実装しました。

✅ 実装・検証済み(ZXingデコードで往復確認)
UPC-A — 実際のGTIN例(03600029145)で符号化→ZXingデコードが完全一致
Code 93 — チェックディジット(C/K)込みで往復確認
ITF (Interleaved 2 of 5) — 往復確認
ITF-14 — 往復確認(GS1チェックディジット込み)
✅ 実装済み(アルゴリズムが自明で表テーブルに依存しない)
Industrial 2 of 5 — 標準的な2-of-5パターンで実装(ZXingは非対応のためデコード検証はできませんが、往復デコードの必要がない単純な重み付け構造です)
Pharmacode — 2進表現そのものなので表テーブル自体が存在せず、実装リスクは低いです
⚠️ 実装したが未検証(要注意)
UPC-E・Codabar — 一通り実装しましたが、ZXingのデコーダで正しく読み取れることを確認できませんでした。UPC-Eはパリティテーブルと開始/終了ガードパターンの全組み合わせ(384通り)を総当たりで試しても一致せず、Codabarも同様に失敗しました。原因は「私の記憶によるテーブルの誤り」か「このzxing-jsビルド自体がUPC-E/Codabarデコードを実質サポートしていない」のいずれか切り分けられませんでした(ZXing自身のエンコーダもこの2方式には非対応というヒントもあり、後者の可能性もあります)。この2つはUIには反映せず、コードのみ残してあります。信頼できる別ツールでの動作確認を推奨します。
