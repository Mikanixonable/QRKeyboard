# リファクタリング計画

このプロジェクトのコードを段階的に整理するための計画。**各フェーズは独立してコミットでき、動作を変えない**ことを原則とする。各フェーズ完了ごとに `node test/encode-decode.test.js` を実行し、UI に触れるフェーズではブラウザで手動確認する(CLAUDE.md 参照)。

## 現状把握 (2026-07-20 調査)

- 合計約 8,500 行。最大は `app.js` (2,758 行、単一 IIFE、関数 77 個)。次いで `qrcode.js` (1,478 行)、`style.css` (1,109 行)。
- `console.log` / `debugger` / TODO コメントの残骸は**なし**。コメントアウトされた死にコードもほぼなし(grep でヒットしたのは日本語の説明コメントのみ)。→ 当初想定の「コメントアウト削除」フェーズは軽微で済む。
- `var` 不使用・`const`/`let` 統一済み。モジュール公開パターン (`module.exports ?? global`) も全ライブラリで統一済み。基礎的なスタイルは良好。
- 主な課題は (1) `app.js` の肥大化と責務混在、(2) マジックナンバーの散在、(3) 描画系関数の重複、(4) 死にコード (UPC-E / Codabar が UI 未接続)。

## フェーズ 1: 死にコード・不要物の整理(低リスク)

1. **UPC-E / Codabar の扱いを決める**: `barcode1d.js` にエンコーダ実装済みだが UI 未接続(ZXing での読み取り未検証のため)。選択肢は「削除」「残して SPEC.md/コメントに未接続の旨を明記(現状維持を明文化)」のどちらか。CLAUDE.md に経緯が書かれているので、**削除ではなくコメントで明示**を推奨。
2. 各ファイルを目視で一巡し、未使用の関数・変数・CSS ルールを洗い出して削除(`style.css` 1,100 行は未使用セレクタが紛れやすい)。
3. 検証: 回帰テスト + 全規格をブラウザで一巡表示。

## フェーズ 2: マジックナンバーの定数化(低リスク)

対象例(調査で確認済みのもの):

- `app.js`: PNG 出力のスケール計算 `2048`, `Math.max(4, Math.min(16, ...))` の 4/16 → `PNG_TARGET_SIZE`, `MIN_SCALE`, `MAX_SCALE` などにファイル冒頭で定義。同じ式が 4 箇所 (1588, 1606, 1700, 1763 行付近) にコピーされているのでヘルパー関数化も同時に行う。
- `app.js`: モバイル判定 `760px` (2720 行) — CSS 側のメディアクエリと同値なので、定数名で「CSS と同期必須」と明記。
- `barcode1d.js` ほか: quiet zone 幅 (9, 10 など) は規格由来の値なので、規格名入りの定数 or コメントで根拠を示す。
- `qrcode.js`: マスク評価のペナルティ係数 (N1〜N4 相当) など、JIS/ISO 規格書の値には規格参照コメントを付ける。**規格由来の数表 (符号表・容量表) は定数化の対象外**(かえって読みにくくなる)。
- 検証: 回帰テストのみで可(値が変わっていなければ出力は同一)。

## フェーズ 3: `app.js` 内の重複排除(中リスク)

1. **PNG 描画 3 兄弟の統合**: `renderFullLinearCanvas` / `renderFullSingleCanvas` / `renderFullMultiCanvas` (1633〜1900 行付近) はスケール計算・余白・キャプション描画のロジックが重複。共通部分を抽出して差分(レイアウト)だけを分ける。
2. 画面描画 (`drawCurrentToCanvas`, 900〜1200 行) と PNG 描画の共通化余地も調査(同一化は無理にしない — 表示用と保存用で要件が違う)。
3. `buildXxxControl` 系 (約 10 関数) の共通パターン (ラベル + セグメントボタン生成) をヘルパーに寄せる。`makeSegButton`/`makeToggle` は既にあるので、その利用徹底が中心。
4. 検証: 回帰テスト + **PNG 保存を各モード(単一/複数/バーコード)で実施しピクセル単位で before/after 比較**(before の PNG をフェーズ開始前に採取しておく)。

## フェーズ 4: `app.js` のモジュール分割(高リスク・任意)

2,758 行の単一 IIFE を責務ごとにファイル分割する。ビルドツールを導入しない方針のため、既存ライブラリと同じ「IIFE + グローバル公開」方式で分割し、`index.html` の script タグ順で依存を解決する。

分割案(関数一覧の調査に基づく):

| 新ファイル | 責務 | 目安 |
|---|---|---|
| `app-controls.js` | `build*Control` 系、バージョン選択タイル、fits 計算キャッシュ | ~600 行 |
| `app-render.js` | canvas 描画 (`drawCurrent*`, キャプション折返し, `renderFull*`) | ~700 行 |
| `app-share.js` | `buildUrlParams` / `loadFromUrl` / 保存・ダウンロード | ~300 行 |
| `app-scan.js` | カメラ/画像デコード、ZXing 連携、色サンプリング | ~450 行 |
| `app.js` | 状態管理・イベント配線・テーマ・レスポンシブ(残り) | ~700 行 |

注意点:

- 分割境界をまたぐ共有状態 (`state` オブジェクト等) の受け渡し設計を先に決める。安易にグローバル変数を増やさない。
- `sw.js` のキャッシュ対象リストと `index.html` の script タグ追加を忘れない(PWA 対応済みのため)。
- このフェーズは効果とリスクのバランスで**実施するか事前に判断**する。フェーズ 1〜3 だけでも保守性はかなり改善する。

## フェーズ 5: テストの拡充(任意・随時)

- 現状テストは `qrcode.js` 系のエンコード/デコードのみ。`barcode1d.js` / `pdf417.js` / `datamatrix.js` / `aztec.js` のエンコード結果(モジュール配列のハッシュ比較など)をゴールデンテストとして追加すると、フェーズ 2〜3 の安全網になる。**理想はフェーズ 2 の前に実施**。

## 進め方の原則

- 1 フェーズ = 1 コミット以上。フェーズをまたぐ変更を混ぜない。
- 動作変更(バグ修正・仕様変更)を見つけても、このリファクタリング中には行わず `dev.md` のチケットとして記録する。
- SPEC.md / CLAUDE.md に影響する変更(ファイル構成変更など)はドキュメントも同一コミットで更新する。
